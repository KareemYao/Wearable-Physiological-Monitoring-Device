## 1. 项目背景与系统总体设计

### 1.1 项目背景

随着可穿戴健康监测设备市场的快速增长，消费者对便携式生命体征（心率、血氧饱和度）监测设备的需求日益迫切。传统医疗级脉搏血氧仪体积大、成本高，难以满足日常佩戴场景。本项目旨在设计一套基于 **反射式光电容积脉搏波（PPG）** 技术的便携式智能心率血氧监测系统，同时集成工业级电源管理模块，实现产品级的完整解决方案。

### 1.2 系统设计目标

| 指标项 | 设计目标 |
|:---|:---|
| 心率测量范围 | 40 ~ 150 BPM |
| 血氧测量范围 | 90% ~ 99% |
| 采样率 | 100 Hz（10ms/次） |
| 数据上报周期 | 2 s |
| 电池容量 | 800 mAh 聚合物锂电池 |
| 电量估算精度 | 温度补偿二维查表法，分辨率 5% |
| 通信接口 | USART1 @ 115200 bps（DMA 发送） |

### 1.3 系统总体架构

系统从功能维度划分为三大子系统：**PPG 信号采集与处理子系统**、**电源管理子系统** 和 **数据通信子系统**。三者通过基于定时器的非阻塞调度框架协同运行于同一 MCU 上。

![系统硬件实物图](E:\data_test\fbbda989a68a248d7041043cc8413046.jpg)

整体数据流路径如下：

```
MAX30102 FIFO ──(软I2C)──▸ PPG原始数据缓存(200采样点)
                               │
                               ▼
                    AC/DC分离 → 滤波 → 峰谷联合检测 → 心率/血氧计算
                               │
                               ▼
                    心率: 环形缓冲区偏差门控校准 ─┐
                    血氧: 直通(无校准)          ─┤→ 滑动平均 → g_vital_signs
                               │
            ┌──────────────────┘
            ▼
    USART1 DMA 数据包发送 ──▸ 上位机
            
ADC CH1(NTC) + CH2(电池) ──(DMA)──▸ 温度查表 → 二维SOC查表 → LED指示
```

## 2. 硬件选型与底层驱动接口

### 2.1 主控 MCU：STM32G030C8T6

本系统选用 ST 公司 Cortex-M0+ 内核的 **STM32G030C8T6** 作为主控芯片，主要考量因素如下：

- **主频**：64 MHz，在超低成本入门级 MCU 中具备充足的实时信号处理算力；
- **超低成本与小封装**：LQFP48 封装，BOM 成本极具竞争力，适合便携式消费产品量产；
- **DMA 控制器**：DMA1 提供 5 个通道，支撑 ADC 采集与 UART 发送的零 CPU 开销传输；
- **定时器资源**：高级定时器 TIM1 提供精确的 10ms 中断基准；
- **资源约束**：64 KB Flash / 8 KB SRAM，要求软件设计必须高度关注内存效率（参见 4.1.2 节 Union 内存复用设计）。

> **无 FPU 说明**：Cortex-M0+ 内核不具备硬件浮点单元，本系统中 SpO2 的 R 值浮点运算和滤波计算均由编译器软浮点库完成。得益于算法中浮点运算量较小（仅在每 2s 批量处理时触发），软浮点开销对 10ms 实时采集任务无可测量的影响。

系统时钟配置采用内部 HSI 16 MHz 经 PLL 倍频后得到 64 MHz 系统主频（PLLM=1, PLLN=8, PLLR=2, 即 16×8/2=64 MHz），APB 总线与系统主频同频运行（无分频），保证 ADC 和 USART 外设运行在最佳时钟域内。该配置见工程中 `SystemClock_Config` 函数的实现。

### 2.2 核心传感器：MAX30102

**MAX30102** 是美信（Maxim）推出的集成红光 LED（660nm）与红外光 LED（880nm）的反射式脉搏血氧传感器，内置 18 位 ADC 和 32 深度 FIFO 缓冲区。

**传感器配置策略**（参考 `HR_Device_Init` 的寄存器写入序列）：

| 寄存器地址 | 写入值 | 功能说明 |
|:---:|:---:|:---|
| `0x0A` | `0x07` | SPO2 配置：ADC量程 4096nA，采样率 100Hz，脉宽 411µs（18bit 分辨率） |
| `0x0C` | `0x18` | 红光 LED 驱动电流精调（约 6.4mA） |
| `0x0D` | `0x18` | 红外光 LED 驱动电流精调（约 6.4mA） |
| `0x09` | `0x03` | 模式控制：SpO2 模式（同时启用红光+红外光通道） |

> **核心调优说明**：LED 驱动电流统一设置为 `0x18`，该值经过反复实验标定，在保证信噪比的前提下最大限度降低功耗与皮肤发热，是平衡信号质量与佩戴舒适度的关键参数。

采样率设为 100Hz，对应 10ms 一次 FIFO 数据更新，与定时器的快任务周期严格对齐，确保无数据丢失。

### 2.3 软件 I2C 通信接口

由于本系统未启用 STM32 的硬件 I2C 外设（项目 HAL 配置中 `HAL_I2C_MODULE_ENABLED` 处于关闭状态），传感器通信采用 **GPIO 模拟软件 I2C** 方案，实现位于 `bsp_soft_i2c.c`。

**引脚分配**：

| 信号 | GPIO | 配置模式 |
|:---:|:---:|:---|
| SCL | **PB3** | 开漏输出 + 上拉 |
| SDA | **PB6** | 开漏输出 + 上拉 |

GPIO 初始化在 `MX_GPIO_Init` 中配置为 `GPIO_MODE_OUTPUT_OD`（开漏模式）并使能内部上拉电阻，符合 I2C 总线电气规范。

软件 I2C 驱动封装了完整的总线协议栈：

- **基本操作**：`I2C_Start`、`I2C_Stop`、`I2C_Send_Byte`、`I2C_Receive_Byte`
- **总线恢复**：`I2C_Bus_Recovery` —— 在通信 NAK 异常时发送 9 个时钟脉冲强制释放总线
- **批量读取**：`My_SI2C_ReadDatas` —— 支持连续读取 MAX30102 FIFO 中的 6 字节数据（3 字节红光 + 3 字节红外光）

时序延迟通过 `I2C_Delay_us` 实现，采用 `volatile` 空循环约 15 次迭代，在 64MHz 主频下约产生 3~5µs 延迟，满足 I2C 标准模式（100kHz）的时序要求。

### 2.4 供电系统与 ADC 检测

系统采用 **800mAh 聚合物锂电池** 供电，通过分压电阻网络将电池电压映射到 MCU 的 ADC 量程范围内。同时，板载贴片式 **NTC 热敏电阻**（负温度系数）用于实时感知环境/电池温度，为电量估算提供温度补偿依据。

**ADC 通道配置**（参考 `MX_ADC1_Init`）：

| 通道 | GPIO | 功能 | 转换顺序 |
|:---:|:---:|:---|:---:|
| ADC_CHANNEL_1 | PA1 | NTC 热敏电阻分压电压 | Rank 1 |
| ADC_CHANNEL_2 | PA2 | 电池端分压电压 | Rank 2 |

ADC 配置为 **12 位分辨率、连续扫描模式、DMA 传输**，采样时间 160.5 个 ADC 时钟周期，数据通过 DMA1 Channel1 自动搬运至内存缓冲区 `BSP_RAM_Rece`（长度 20，交错存储 10 组双通道数据），全程无 CPU 干预。

## 3. 核心软件架构

### 3.1 "快慢分离"非阻塞调度机制

本系统的核心软件架构基于 **定时器中断驱动的"快慢分离"非阻塞调度模型**，避免使用操作系统（Bare-Metal），在资源受限的嵌入式环境下实现任务的确定性调度。

**调度框架核心**位于 `HAL_TIM_PeriodElapsedCallback`，由高级定时器 TIM1 提供中断基准。

#### TIM1 配置（参考 `MX_TIM1_Init`）

| 参数 | 值 | 说明 |
|:---|:---:|:---|
| 预分频器 Prescaler | 63 | 64MHz APB 时钟分频后得 1MHz 计数频率 |
| 自动重装载值 Period | 999 | 1MHz / 1000 = 1kHz → 1ms 单次溢出 |
| 重复计数器 RepetitionCounter | 9 | 每 10 次溢出触发一次更新中断 |
| **实际中断周期** | **10ms** | 1ms × 10 = 10ms |

> TIM1 为高级定时器，具备重复计数器（RCR）特性。利用该特性将 1ms 基础定时扩展为 10ms 中断周期，无需额外的软件计数器，硬件层面即保证了精确的中断间隔。

#### 快任务（10ms 周期）

每 10ms 中断触发一次，执行传感器数据的**实时采集**任务：

```
每10ms中断:
  1. 通过软件 I2C 读取 MAX30102 FIFO 的 6 字节数据
  2. 调用 MAX30102_ProcessData() 将数据送入 PPG 处理流水线
     ├── 缓存至 200 点批量缓冲区
     └── 缓冲区满时触发: 滤波 → 峰谷联合检测 → 心率/血氧计算 → 心率校准 → 滑动平均
```

快任务的核心诉求是**零丢失**：MAX30102 在 100Hz 采样率下，FIFO 每 10ms 更新一组新数据。若 CPU 未能及时读取，FIFO 溢出将导致波形断裂，从而使心率检测算法失效。因此快任务必须与传感器采样率严格同步。

#### 慢任务（2s 周期）

在快任务回调内部通过软件计数器实现 2s 周期：

```
快任务内: data_time_counter++
当 data_time_counter >= 200 时 (200 × 10ms = 2s):
  1. 重置计数器
  2. 调用 Send_DataPacket() —— 构造协议帧并通过 USART1 DMA 发送
  3. 调用 BSP_ADC_StartDMA() —— 启动 ADC DMA 采集（后台处理由 main 循环完成）
```

慢任务承载**非实时性**操作：数据上报和电量监测对时间精度要求较低（秒级），因此以 2s 为周期执行，避免频繁串口发送和 ADC 后处理占用宝贵的中断时间片。

### 3.2 DMA 数据通信子系统

数据上报采用 **USART1 + DMA 发送** 的零拷贝发送架构，实现位于 `bsp_usart_dma.c`。

**协议帧格式**（6 字节定长帧）：

| 字节序号 | 0 | 1 | 2 | 3 | 4 | 5 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| 字段 | 帧头1 | 帧头2 | 心率 | 血氧 | 校验和 | 帧尾 |
| 值 | `0xAA` | `0x55` | HR | SpO2 | SUM[0:3] | `0xBB` |

发送流程引入 **`tx_complete` 标志位**（参考 `UART_Transmit_DMA`）实现发送互斥保护：仅当上一帧 DMA 传输完成（由 `HAL_UART_TxCpltCallback` 回调置位）后，才允许启动新一帧的传输，防止在中断上下文中发生 DMA 描述符覆盖。

### 3.3 初始化流程

系统上电后的初始化序列定义在 `main` 函数中，严格遵循**外设依赖顺序**：

```
HAL_Init() → SystemClock_Config()
  → MX_GPIO_Init()          // GPIO 必须最先初始化（I2C引脚依赖）
  → MX_DMA_Init()           // DMA 控制器必须先于使用 DMA 的外设
  → MX_ADC1_Init()          // ADC + DMA 链路建立
  → MX_TIM1_Init()          // 定时器配置（但尚未启动）
  → MX_USART1_UART_Init()   // UART + DMA 链路建立
  → HR_Device_Init()        // 通过软I2C写入MAX30102寄存器配置
  → BSP_TIM_Start_IT()      // 启动定时器中断 ——系统开始运行
```

> **关键设计点**：`MX_DMA_Init` 必须在 `MX_ADC1_Init` 和 `MX_USART1_UART_Init` 之前调用。因为 ADC 和 USART 的 MSP 初始化回调中会配置各自的 DMA 数据流，若 DMA 控制器时钟未提前使能，将导致 DMA 初始化失败。

## 4. 核心算法解析

### 4.1 PPG 信号处理与心率血氧计算（bsp_ir_red_cal2.c）

`bsp_ir_red_cal2.c` 是本系统的算法核心，实现了从 MAX30102 原始 FIFO 数据到最终心率（HR）和血氧饱和度（SpO2）输出的完整信号处理流水线。

PPG波形图

![PPG 原始波形采集（绿色：IR 红外通道，橙色：RED 红光通道）](images/ppg_waveform.png)

#### 4.1.1 数据采集与批量缓存策略

系统采用 **"逐点采集、批量处理"** 的双阶段架构（实现于 `MAX30102_CollectAndProcessAC` 静态函数）。

**第一阶段——逐点缓存**：每次定时器中断调用时，解析 6 字节 FIFO 数据为红光和红外光的 18 位 ADC 值（高 18 位有效，通过 `& 0x3FFFF` 掩码提取），并依次存入长度为 200 的原始数据缓冲区 `raw_red_buf` / `raw_ir_buf`。

**第二阶段——批量触发**：当缓冲区累积满 200 个采样点（对应 200 × 10ms = 2s 的连续波形数据）时，触发后续全部信号处理步骤。该设计确保了算法拥有足够长度的连续波形窗口用于可靠的波峰检测。

> **设计理由**：静息状态下人体心率范围为 40~100 BPM，对应心搏周期 0.6~1.5s。2s 窗口至少可捕获 1~3 个完整心搏周期，满足双波峰检测的最低要求。

#### 4.1.2 Union 联合体内存复用设计（ReusableMem_t）

在资源受限的嵌入式 MCU 上（STM32G030 的 SRAM 仅有 8 KB），200 点批量处理所需的多组大型缓冲区对 RAM 预算构成严峻挑战。V2.0 版本引入了基于 **C 语言 Union 联合体**的内存复用机制，将处理流水线中 **生命周期不重叠** 的两组缓冲区映射到同一块物理内存。

**联合体定义**（`bsp_ir_red_cal2.c`）：

```c
typedef union {
    struct {  /* 阶段A：采集与滤波 */
        uint32_t raw_red[BATCH_BUF_SIZE];       /* 200 × 4B = 800B */
        uint32_t raw_ir[BATCH_BUF_SIZE];        /* 200 × 4B = 800B */
        uint32_t red_filtered[BATCH_BUF_SIZE];  /* 200 × 4B = 800B */
        uint32_t ir_filtered[BATCH_BUF_SIZE];   /* 200 × 4B = 800B */
    } stage_filter;                             /* 合计: 3200 字节 */

    struct {  /* 阶段B：寻峰与寻谷 */
        uint8_t  pos_idx[BATCH_BUF_SIZE];       /* 200 × 1B = 200B */
        float    pos_val[BATCH_BUF_SIZE];       /* 200 × 4B = 800B */
        uint8_t  neg_idx[BATCH_BUF_SIZE];       /* 200 × 1B = 200B */
        float    neg_val[BATCH_BUF_SIZE];       /* 200 × 4B = 800B */
    } stage_peaks;                              /* 合计: 2000 字节 */
} ReusableMem_t;

static ReusableMem_t reuse_mem;
```

**生命周期分析与安全性论证**：

两组缓冲区的使用时序严格遵循 **"先A后B、不交叠"** 的顺序约束：

1. **阶段A（采集与滤波）**：`raw_red` / `raw_ir` 在逐点缓存阶段被写入原始 ADC 值；缓冲区满后，三点滑动平均滤波将结果写入 `red_filtered` / `ir_filtered`，随后立即从滤波结果中提取 DC 值并计算 AC 分量（存入独立的 `red_ac` / `ir_ac` 浮点数组）。**至此，阶段A的四个 `uint32_t` 数组已完成全部使命，其中数据不再被任何后续环节引用。**

2. **阶段B（寻峰与寻谷）**：`MAX30102_DetectPeaks` 函数在进入前首先 `memset` 清零整个 `stage_peaks` 结构体，然后遍历 `ir_ac` 浮点数组（注意：`ir_ac` 是独立分配的 `static float` 数组，不在 Union 内），将正值候选点的索引和幅值分别写入 `pos_idx` / `pos_val`，负值候选点写入 `neg_idx` / `neg_val`。

**两阶段之间存在天然的数据流断点**：阶段A的输出是浮点 AC 数组（存放在 Union 外部），阶段B的输入也是该浮点 AC 数组——阶段B完全不依赖阶段A的原始/滤波缓冲区。因此两组缓冲区可以安全地共用同一块物理内存，不存在数据竞争或踩踏风险。

**定量 RAM 节省分析**：

| 方案 | stage_filter | stage_peaks | 总占用 |
|:---:|:---:|:---:|:---:|
| 独立分配 | 3200 B | 2000 B | **5200 B** |
| Union 复用 | — | — | **3200 B**（取两者较大值） |
| **节省** | — | — | **2000 B（38.5%）** |

> **工程价值**：对于 SRAM 仅 8 KB 的 STM32G030 而言，2000 字节的节省相当于整个 RAM 空间的 **25%**。这一设计直接决定了该算法能否在目标芯片上运行——若不进行内存复用，仅此一个模块的缓冲区开销就将占用 SRAM 的 63.5%（5200/8192），加上栈空间、全局变量和 HAL 库开销后极有可能溢出。Union 复用将该比例压缩至 39%（3200/8192），为系统其余模块留出了必要的内存裕量。

#### 4.1.3 三点滑动平均滤波

原始 PPG 信号易受运动伪影和高频噪声干扰。系统采用 **窗口大小为 3 的滑动平均滤波器**（`FILTER_WINDOW = 3`）对 200 点原始数据进行平滑处理：

$$
y[n] = \frac{x[n-1] + x[n] + x[n+1]}{3}, \quad n = 1, 2, \ldots, N-2
$$

该滤波器本质上是一个低通 FIR 滤波器，其 -3dB 截止频率约为采样率的 1/6，即约 16.7Hz，在保留心率信号主频成分（0.67~2.5Hz）的同时有效抑制高频毛刺。

滤波后的有效数据长度为 N - 2 = 198 点，分别存入 `red_filtered` 和 `ir_filtered` 数组。

#### 4.1.4 AC/DC 分离

PPG 信号的物理组成为：

$$
\text{PPG}(t) = \underbrace{\text{DC}}_{\text{组织/静脉/恒定光}} + \underbrace{\text{AC}(t)}_{\text{动脉搏动分量}}
$$

其中 DC 分量（直流分量）反映组织、静脉血液和恒定背景光的吸收，AC 分量（交流分量）反映动脉搏动引起的光吸收变化，是提取心率和血氧的有效信号。

**DC 提取**：对滤波后 198 个数据点求算术平均：

$$
\text{DC}_{\text{red}} = \frac{1}{N-2} \sum_{i=0}^{N-3} \text{red\_filtered}[i], \quad
\text{DC}_{\text{ir}} = \frac{1}{N-2} \sum_{i=0}^{N-3} \text{ir\_filtered}[i]
$$

**AC 分离**：逐点减去全局 DC 值，得到以零为中心的交流波形：

$$
\text{AC}_{\text{red}}[i] = \text{red\_filtered}[i] - \text{DC}_{\text{red}}, \quad
\text{AC}_{\text{ir}}[i] = \text{ir\_filtered}[i] - \text{DC}_{\text{ir}}
$$

分离后的 `red_ac` 和 `ir_ac` 数组即为后续波峰检测和 SpO2 计算的输入。

#### 4.1.5 基于双极值提取的峰谷联合心率检测算法

V2.0 版本对心率检测算法进行了彻底重构，摒弃了旧版"一阶导数主检测 + 负谷兜底搜索"的双路径架构，转而采用基于 **`ExtractTwoExtremes` 双极值提取** 的 **峰谷联合** 心率计算方案。该算法从 IR 红外光 AC 信号中同时提取正峰和负谷两组独立心搏周期估计，通过双路交叉验证显著提升了对噪声和伪影的鲁棒性。

##### （1）核心函数：ExtractTwoExtremes

`ExtractTwoExtremes` 是新算法的基础构件，其功能是从一组候选极值点中提取满足最小时间间隔约束的两个最显著极值。该函数通过 `is_positive` 参数切换工作模式——正峰提取或负谷提取。

**算法步骤**：

```
ExtractTwoExtremes(is_positive, ext_idx[], ext_val[], out_peaks[]):

Step 1 - 候选集构建:
  遍历 ir_ac[0 ~ N-3]:
    若 is_positive 且 ir_ac[i] > 0 → 记录 (i, ir_ac[i]) 为正峰候选
    若 !is_positive 且 ir_ac[i] < 0 → 记录 (i, ir_ac[i]) 为负谷候选

Step 2 - 全局极值选取:
  在整个候选集中找出幅值最大的点 ext1（正峰模式取最大正值,
  负谷模式取最小负值）

Step 3 - 次全局极值选取（带间隔约束）:
  在候选集中排除 ext1, 遍历所有剩余候选点:
    若 |idx[k] - idx[ext1]| >= MIN_PEAK_INTERVAL:
      记录满足间隔约束且幅值最大的点为 ext2

Step 4 - 按时序排列输出:
  out_peaks[0] = min(ext1, ext2) 的时间索引
  out_peaks[1] = max(ext1, ext2) 的时间索引
```

**为什么选取"最大 + 次最大"而非全部极值？**

PPG 信号在 2s 窗口内通常包含 1~3 个完整心搏周期，但信号中不可避免地混杂着高频噪声引起的毛刺极值、运动伪影造成的虚假波峰以及传感器接触不良导致的基线漂移。若尝试提取所有极值后再进行配对，将面临 $O(n^2)$ 的组合爆炸问题——从数十个候选极值中自动判定哪两个属于相邻心搏本身就是一个高维优化问题。

**"全局最大值 + 间隔约束次最大值"策略**的核心洞见是：**真正的心搏极值在幅值排序中几乎总是位列前茅**。生理性的动脉搏动分量是 AC 信号中能量最集中的成分，其波峰/波谷幅值远大于随机噪声。因此，选取幅值最大的两个极值（而非全部极值中的任意两个）可以以 $O(n)$ 的线性复杂度高概率锁定真实心搏对应的两个极值点，天然实现了大幅度噪声过滤。

##### （2）峰谷联合检测流程

`MAX30102_DetectPeaks` 函数对正峰和负谷各执行一次 `ExtractTwoExtremes`，输出 4 个极值索引：

```c
void MAX30102_DetectPeaks(uint16_t *peaks)
{
    memset(&reuse_mem.stage_peaks, 0, sizeof(reuse_mem.stage_peaks));

    /* 提取正峰: peaks[0], peaks[1] */
    ExtractTwoExtremes(1, pos_peak_idx, pos_peak_val, &peaks[0]);

    /* 提取负谷: peaks[2], peaks[3] */
    ExtractTwoExtremes(0, neg_valley_idx, neg_valley_val, &peaks[2]);
}
```

由此得到两组独立的心搏周期估计：

$$
\text{pos\_interval} = \text{peaks}[1] - \text{peaks}[0] \quad \text{（峰峰间隔）}
$$

$$
\text{neg\_interval} = \text{peaks}[3] - \text{peaks}[2] \quad \text{（谷谷间隔）}
$$

分别计算两路心率：

$$
\text{hr\_pos} = \frac{60 \times f_s}{\text{pos\_interval}}, \quad
\text{hr\_neg} = \frac{60 \times f_s}{\text{neg\_interval}}, \quad f_s = 100\text{Hz}
$$

##### （3）MIN_PEAK_INTERVAL 间距约束的物理含义

`MIN_PEAK_INTERVAL = 30` 个采样点（对应 30 × 10ms = 0.3s）设定了两个合法极值之间的最小时间间隔，其物理含义是 **一个完整心搏周期的下限**。

在 PPG 波形中，单个心搏周期内部呈现复杂的多峰结构：收缩期主峰、重搏切迹（dicrotic notch）以及重搏波（dicrotic wave）。若不施加间隔约束，`ExtractTwoExtremes` 可能将同一心搏内的主峰与重搏波误判为两个相邻心搏的峰值——此时计算出的"心率"将是真实心率的 2 倍以上。

0.3s 的下限对应心率上限 200 BPM（$60/0.3 = 200$），充分覆盖了从运动到极端心动过速的全部生理场景。同时，该约束不会误伤正常心率：即使心率高达 180 BPM（心搏周期 0.33s），两个相邻真正的心搏极值间隔仍大于 0.3s，不会被阈值截断。

##### （4）峰值法与谷值法互为验证的工程意义

将正峰法（hr_pos）和负谷法（hr_neg）作为两条独立估计路径，是 V2.0 算法相对于旧版的核心升级。其工程价值体现在以下三个维度：

| 维度 | 正峰法（hr_pos） | 负谷法（hr_neg） | 联合优势 |
|:---|:---|:---|:---|
| **信号特征** | 检测收缩期快速上升沿对应的正向极值 | 检测舒张期末端的负向极值 | 互为补充，覆盖完整心动周期 |
| **噪声敏感区域** | 易受高频毛刺影响（正峰区域毛刺多） | 对高频噪声较不敏感（谷底相对平滑） | 一路失效另一路兜底 |
| **运动伪影表现** | 运动伪影可能淹没正峰（基线上抬） | 运动伪影通常不改变谷底形态 | 抗运动干扰能力倍增 |

在后续的 `MAX30102_CalcVitalSigns` 中，系统对两路心率进行 **联合校验**：当两路偏差在合理范围内时取其平均值作为最终心率；当仅一路可用时降级使用单路结果；当两路均异常时拒绝输出——这一"冗余估计 + 择优融合"策略本质上是航空航天领域 **三模冗余（TMR）** 思想在低成本消费设备上的工程简化版本。

#### 4.1.6 心率与血氧饱和度计算

峰谷检测完成后，心率和血氧的计算由 `MAX30102_CalcVitalSigns` 函数完成。

**心率计算**：

$$
\text{hr\_pos} = \frac{60 \times f_s}{\text{peaks}[1] - \text{peaks}[0]}, \quad
\text{hr\_neg} = \frac{60 \times f_s}{\text{peaks}[3] - \text{peaks}[2]}
$$

其中 $f_s = 100$ Hz 为采样率。有效心率范围为 [40, 150] BPM，超出此范围的结果视为无效，不予采纳。

**两路心率联合选择策略**：

```
判断 hr_pos 和 hr_neg 是否分别在 [40, 150] 范围内:
  ├── 仅 hr_pos 有效 → 使用 hr_pos
  ├── 仅 hr_neg 有效 → 使用 hr_neg
  ├── 两路均有效    → 取 (hr_pos + hr_neg) / 2 平均
  └── 两路均无效    → 维持上次输出值不变
```

**血氧饱和度（SpO2）计算**：

采用经典的 **比值比（Ratio of Ratios）** 公式：

$$
R = \frac{\text{AC}_{\text{red\_max}} \times \text{DC}_{\text{ir}}}{\text{AC}_{\text{ir\_max}} \times \text{DC}_{\text{red}}}
$$

$$
\text{SpO2} = 104 - 17 \times R \quad (\%)
$$

其中 AC_red_max 和 AC_ir_max 分别为红光和红外光 AC 分量在整个 2s 窗口内的峰值。

> **经验公式说明**：SpO2 = 104 - 17R 是基于 Beer-Lambert 光吸收定律推导并经大量受试者数据回归得到的标准线性近似公式。系数 104 和 17 代表了氧合血红蛋白与脱氧血红蛋白在 660nm/880nm 双波长下的摩尔消光系数差异。

**硬限幅保护**：计算结果进行边界钳位处理——SpO2 低于 90% 时锁定为 90%，高于 100% 时限制为 99%，防止不合理值传递至上位机。

#### 4.1.7 心率校准机制（环形缓冲区 + 偏差门控）

原始的逐批次心率计算结果存在固有的波动性（波峰检测的随机抖动、运动干扰等），不适合直接作为最终输出。V2.0 版本设计了基于 **环形缓冲区 `smooth_buf` + 偏差门控** 的两阶段校准状态机，以极低的 RAM 开销（仅 5 字节）实现稳健的心率输出平滑。

> **为什么校准机制仅覆盖心率而不包含血氧？** SpO2 由红光与红外光 AC/DC 比值公式直接计算，其数值本身具有较高的稳定性；而心率依赖峰间距检测，对噪声和波形畸变敏感得多。实测表明 SpO2 仅需后级滑动平均滤波器（窗口=5）即可获得良好的降噪效果。将两者分别处理还避免了一个潜在耦合缺陷：若心率与血氧共用同一 index 推进逻辑，当心率偏差合格而血氧偏差超限时，index 前进但血氧槽位未写入有效值（残留 0），后续基准计算将被 0 值污染。

**核心数据结构**：

```c
#define SMOOTH_BUF_SIZE  3

static struct {
    uint8_t index;                          /* 当前写入位置 (0 ~ SMOOTH_BUF_SIZE-1) */
    uint8_t averaged;                       /* 是否已完成首次校准 (0/1) */
    uint8_t heart_rate[SMOOTH_BUF_SIZE];    /* 心率环形缓冲 */
} smooth_buf;
```

##### 阶段一：首次校准锚点建立（averaged == 0）

系统冷启动后，`averaged` 标志为 0，表示尚未建立可信的基准值。此时系统**拒绝一切非静息数据**，等待捕获第一组"完美静息数据"作为后续所有偏差判断的基准锚点。

**静息合理性判据** `quiet_is_reasonable`：

$$
\text{quiet\_is\_reasonable} = \begin{cases}
1 & \text{if } 60 < \text{hr\_pos} < 90 \;\wedge\; 60 < \text{hr\_neg} < 90 \\
0 & \text{otherwise}
\end{cases}
$$

> **为什么要求正峰和负谷两路心率同时落入 60~90 BPM？** 这一严苛条件确保首次锚点具有极高的可信度：两条独立检测路径给出一致且处于静息区间的心率，排除了传感器尚未贴合稳定、手指按压力度不当等启动阶段常见干扰的可能性。

首次捕获到满足条件的数据后，将两路心率的均值 `(hr_pos + hr_neg) / 2` 写入 `smooth_buf.heart_rate[0]`，同时将 `averaged` 置 1、`index` 推进至 1——系统正式进入心率稳态校准阶段。

##### 阶段二：滑动窗口周期刷新（averaged == 1）

建立锚点后，每 2s 产生一组新的心率数据，系统对其执行 **偏差门控** 后择优写入环形缓冲区：

**偏差门控规则**：

| 指标 | 允许偏差 | 超限处理 |
|:---:|:---:|:---|
| hr_pos 或 hr_neg 与上一次存入的心率值 | ≤ 10 BPM | 至少一路偏差合理即接受 |
| 两路心率均超过偏差门限 | — | **拒收整组数据**，维持上次稳态值 |

**心率选择器**（在偏差合理时的进一步决策）：

```
若仅 hr_pos 偏差 ≤ 10 → 使用 hr_pos
若仅 hr_neg 偏差 ≤ 10 → 使用 hr_neg
若两路偏差均 ≤ 10     → 使用 (hr_pos + hr_neg) / 2
```

**环形缓冲与周期性基准刷新**：

数据写入后 `index` 自增。当 `index` 回绕至 0（即一个完整窗口 `SMOOTH_BUF_SIZE = 3` 组数据采集完毕）时，系统自动计算窗口内所有数据的算术平均值，覆盖写入 `smooth_buf[0]` 作为新一轮基准锚点：

$$
\text{smooth\_buf.heart\_rate}[0] = \frac{1}{S} \sum_{i=0}^{S-1} \text{smooth\_buf.heart\_rate}[i], \quad S = \text{SMOOTH\_BUF\_SIZE}
$$

这一机制使系统能够跟踪心率的缓慢漂移（如从静息过渡到轻度活动），而不被短时突变污染。

> **设计哲学**：相比旧版的 5 容量缓冲区 + ΔHR ≤ 20 的宽松偏差阈值，V2.0 将缓冲区缩小至 3、偏差阈值收紧至 ΔHR ≤ 10。更小的缓冲区意味着更快的基准更新频率（6s 一轮 vs 旧版 10s），更严格的门控意味着更低的噪声通过率——以略微牺牲对快速心率变化的跟踪速度为代价，换取了输出的极致稳定性。同时，V2.0 将血氧从校准环移出，仅依赖后级滑动平均滤波（窗口=5）降噪，消除了心率-血氧 index 耦合带来的基准污染风险。在面向健康监测的消费场景下，这一权衡是合理的：用户更期望看到平稳过渡的数值，而非剧烈跳变的"真实值"。

**第二级：滑动平均滤波器**（窗口大小 = 3）

经过第一级环形缓冲区校准后的心率值，通过 `HR_SlidingAvgFilter`（窗口=3）进行滑动平均；血氧值不经过校准环，直接由 `SPO2_SlidingAvgFilter`（窗口=5）进行滑动平均降噪：

$$
\text{smooth\_hr} = \frac{1}{\min(n, W_{hr})} \sum_{i=0}^{\min(n,W_{hr})-1} \text{hr\_buf}[i], \quad W_{hr}=3
$$

$$
\text{smooth\_spo2} = \frac{1}{\min(n, W_{spo2})} \sum_{i=0}^{\min(n,W_{spo2})-1} \text{spo2\_buf}[i], \quad W_{spo2}=5
$$

两个滤波器均在系统启动初期（有效值不足窗口大小时）自动降阶运行，避免零值污染平均结果。血氧采用更大的窗口（5 vs 3）以补偿其不经过校准环的事实，实测中该配置在稳定性与响应速度之间取得了良好平衡。

**第三级：全局输出结构体**

最终平滑值写入全局结构体 `g_vital_signs`（类型 `FilteredVitalSigns`），供数据通信子系统的 `Send_DataPacket` 读取并打包发送。

### 4.2 基于 DMA 中断与前后台分离的事件驱动架构（bsp_adc_dma.c + main.c）

在嵌入式系统中，**ADC 数据后处理**（浮点除法、多维查表、LED GPIO 操作）的执行开销远大于一次简单的寄存器读写。若将此类耗时逻辑直接放置于硬件中断服务程序（ISR）中，将导致中断上下文长时间被占用，进而阻塞 TIM1 10ms 定时中断的响应——其直接后果是 MAX30102 的软件 I2C 读取被延迟，PPG 波形数据出现不可恢复的 **断层与丢点**，心率检测算法将彻底失效。

为从根本上规避此风险，本系统在电源管理子系统中设计了一套极其严格的 **"前后台分离"事件驱动架构**，将硬件触发与数据处理在时间域上彻底解耦。

#### 4.2.1 前台（硬件中断层）：快进快出，仅置标志

电源管理的 ADC 采集链路采用 **"按需单次启动"** 而非常驻循环的策略。其触发时序如下：

1. **定时器 2s 慢任务触发**：当 `bsp_tim.c` 中的 2s 周期计数器溢出时，调用 `BSP_ADC_StartDMA()` 瞬间启动 **ADC1 + DMA** 硬件链路，发起一轮 20 个数据（10 组 × 双通道交错）的连续扫描转换；

2. **DMA 传输完成中断**：当 DMA 搬运完最后一个 `uint16_t` 数据后，硬件自动触发 `HAL_ADC_ConvCpltCallback` 回调。在该回调函数内部，系统执行且 **仅执行** 以下两步操作：

```c
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef* hadc)
{
    if(hadc->Instance == ADC1)
    {
        HAL_ADC_Stop_DMA(hadc);   // ① 立即关闭 ADC 硬件，杜绝空转耗电
        adc_data_ready = 1;       // ② 置位轻量级标志位（1 条 ARM 指令）
    }                             // → 立即返回，总耗时 < 5µs
}
```

> **设计哲学**：DMA 完成回调属于 **DMA 硬件中断**，其优先级与 TIM1 更新中断处于同一抢占等级。若在此回调中直接执行浮点运算和查表逻辑（耗时可达数百微秒），将与 10ms 周期的心率采集任务形成临界竞争。因此，回调函数严格遵循 **"快进快出"** 原则——仅完成 ADC 硬件关停和标志位置位，整体执行时间控制在微秒级别，对心率采集任务的中断延迟影响趋近于零。

#### 4.2.2 后台（主循环层）：空闲轮询，从容处理

所有耗时的数据后处理逻辑被严格下沉至 `main.c` 的 `while(1)` **超级循环**中：

```c
while(1) {
    if (adc_data_ready == 1) {
        Task_Battery_Monitor();           // 执行完整的电源管理流水线:
          // ├── BSP_ADC_Process()        //   10 组 ADC 数据均值 + 浮点电压转换
          // ├── Get_Temperature()        //   NTC 一维查表 → 环境温度
          // ├── Get_Battery_Capacity()   //   二维查表 → SOC 百分比
          // └── LED 状态驱动            //   GPIO 电平翻转
        adc_data_ready = 0;               // 清除标志位
    }
}
```

主循环的执行处于所有中断的 **最低优先级**，即使 `Task_Battery_Monitor()` 执行耗时达到毫秒级别，也可被 TIM1 中断随时抢占。当 10ms 心率采集中断到来时，CPU 立即暂停电量处理、跳转执行 I2C 读取，待心率数据安全入缓冲区后再返回继续未完成的查表运算——整个过程对 PPG 波形的连续性实现了 **零影响**。

#### 4.2.3 架构对比与工程价值

| 对比维度 | 全中断方案（反面） | 前后台分离方案（本系统） |
|:---|:---|:---|
| ADC 后处理执行上下文 | DMA 完成中断内 | main() 超级循环内 |
| 对心率 10ms 中断的影响 | 阻塞数百µs，波形断层 | 零阻塞，可被随时抢占 |
| ADC 硬件空转 | 持续运行，浪费电流 | 按需启停，采完即关 |
| 系统耦合度 | 中断嵌套，难以调试 | 前后台解耦，职责清晰 |

> **架构总结**：该设计本质上是经典嵌入式 **"前后台系统"（Foreground/Background System）** 在多外设协同场景下的标准实践。前台（ISR）仅负责硬件事件的捕获与标志通知，后台（超级循环）承担所有可延迟的计算密集型任务。此架构在不引入 RTOS 的前提下，以极低的代码复杂度实现了确定性的实时响应保障。

### 4.3 基于温度补偿的二维查表法 SOC 电量估算（bsp_adc_dma.c）

`bsp_adc_dma.c` 实现了完整的电源管理算法，核心是一套 **基于温度补偿的二维查表法（2D Look-up Table）** SOC（State of Charge）电量估算方案。

#### 4.3.1 为什么需要温度补偿？

锂电池的放电特性受温度影响显著。同一节电池在 -10°C 和 30°C 下，相同端电压对应的剩余电量可能相差 30% 以上。这是因为：

- **低温时**：锂离子在电解液中的扩散速率降低，电池内阻急剧增大，导致端电压在中低电量段快速跌落——同样 3.6V 的端电压，在低温下对应的真实剩余电量远低于常温下的值。
- **高温时**：电化学反应活性增强，放电平台更加平坦，电压-电量映射关系更接近理想状态。

因此，**仅凭电池端电压无法准确估算剩余电量**，必须引入温度作为第二维度进行联合判定。这正是工业级电池管理系统（BMS）中普遍采用的设计思路。

#### 4.3.2 双通道 ADC 数据处理

ADC DMA 循环采集的 20 个 `uint16_t` 数据以 **通道交错方式** 存储在 `BSP_RAM_Rece` 缓冲区中：

```
索引:  [0]   [1]   [2]   [3]   [4]   [5]  ...  [18]  [19]
通道:  CH1   CH2   CH1   CH2   CH1   CH2  ...   CH1   CH2
       NTC   BAT   NTC   BAT   NTC   BAT  ...   NTC   BAT
```

调用 `BSP_ADC_Process` 时，对 10 组数据分通道累加取平均，再转换为电压值：

$$
V_{\text{NTC}} = \frac{\text{ADC\_AVG\_CH1}}{4095} \times 3.3\text{V}
$$

$$
V_{\text{BAT}} = \frac{\text{ADC\_AVG\_CH2}}{4095} \times 3.3\text{V} \times 2
$$

> **电池通道 ×2 系数说明**：电池端电压（3.3V~4.2V）超出 ADC 量程（0~3.3V），因此硬件上采用 1:1 等比分压电阻网络将电池电压减半后送入 ADC，软件端乘以 2 还原真实电压。

得到 **NTC 分压电压**（`Ther_V`）和 **电池真实电压**（`Bat_V`）两个关键物理量。

#### 4.3.3 一维温度查表

NTC（负温度系数）热敏电阻的阻值与温度呈指数反比关系。系统预置了 **7 个温度节点**（-10°C 至 50°C，间隔 10°C）对应的分压电压值：

| 温度 (°C) | -10 | 0 | 10 | 20 | 30 | 40 | 50 |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| NTC 分压电压 (V) | **3.24** | **3.20** | **3.14** | **3.06** | **2.94** | **2.78** | **2.58** |

> **V2.0 电路变更说明**：相比旧版电路（电压节点范围约 0.60V~2.80V），新版 NTC 检测电路的电压节点整体上移至 2.58V~3.24V，逼近 VCC（3.3V）。这一变化源于 **固定分压电阻值的大幅缩小**（从旧版约 12kΩ 降至新版约 1kΩ），具体分析见 4.3.5 节。

NTC 检测电路采用 **上拉分压拓扑**：固定电阻接于 VCC 和 ADC 采样节点之间（上拉），NTC 热敏电阻接于 ADC 采样节点和 GND 之间（下拉）。分压电压的解析表达式为：

$$
V_{\text{ADC}} = V_{\text{CC}} \times \frac{R_{\text{NTC}}(T)}{R_{\text{NTC}}(T) + R_{\text{fixed}}} = 3.3 \times \frac{R_{\text{NTC}}(T)}{R_{\text{NTC}}(T) + 1000} \quad (\text{V})
$$

由于 $R_{\text{NTC}}$ 在 -10°C~50°C 范围内约为 4kΩ~42kΩ，远大于 $R_{\text{fixed}} = 1\text{k}\Omega$，因此分压输出电压普遍在 2.5V 以上、接近 VCC——温度越高，$R_{\text{NTC}}$ 越小，分压电压越低，但幅度受限于 $R_{\text{fixed}}$ 的"压缩效应"。

`Get_Temperature` 函数遍历上述查表数组，采用 **就近匹配** 策略：在相邻两个节点之间计算当前电压到两端点的差值，选取距离更近的节点温度作为输出。同时对超出量程的输入进行边界钳位处理。

**查表算法伪代码**：

```
输入: current_voltage
若 voltage >= V[-10°C]（3.24V）→ 返回 -10°C (极低温/断线)
若 voltage <= V[50°C] （2.58V）→ 返回 50°C  (极高温/短路)

遍历相邻节点对 (T[i], T[i+1]):
  若 V[i] >= voltage > V[i+1]:
    diff1 = V[i] - voltage
    diff2 = voltage - V[i+1]
    返回 diff1 < diff2 ? T[i] : T[i+1]
```

#### 4.3.4 二维查表法 SOC 估算

核心在于一张 **7 行 × 6 列的二维电量映射表** `Battery_Capacity_Table`（定义于 `bsp_adc_dma.c`）：

- **行维度**（7 行）：对应 7 个温度节点（-10°C 至 50°C）
- **列维度**（6 列）：对应 6 个电池电压节点（4.2V 至 3.3V）
- **表格元素**：该温度和电压条件下的电量百分比（0~100%）

| 温度 \ 电压 | 4.2V | 4.0V | 3.8V | 3.6V | 3.5V | 3.3V |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **-10°C** | 90 | 50 | 15 | 0 | 0 | 0 |
| **0°C** | 95 | 70 | 40 | 10 | 0 | 0 |
| **10°C** | 100 | 80 | 55 | 20 | 5 | 0 |
| **20°C** | 100 | 85 | 60 | 25 | 10 | 0 |
| **30°C** | 100 | 90 | 65 | 30 | 10 | 0 |
| **40°C** | 100 | 90 | 65 | 30 | 10 | 0 |
| **50°C** | 100 | 90 | 65 | 30 | 10 | 0 |

**表格设计逻辑解读**：

- **同一电压，低温电量更低**：以 3.8V 为例，-10°C 时仅剩 15%，而 30°C 时仍有 65%——这正是低温导致内阻增大、端电压虚高的体现；
- **满电电压 4.2V**：低温下标定为 90% 而非 100%，反映了极端低温下即使刚充满电，可用容量也已衰减约 10%；
- **-10°C 行的激进保守策略**：V2.0 相比旧版在低温行进一步压低了中低压段的电量值（如 4.0V 列从 70% 降至 50%，3.8V 列从 40% 降至 15%，3.6V 及以下直接归零），反映了最新电池低温实测数据中**容量衰减幅度超出预期**的修正——宁可低估电量导致用户提前充电，也不高估电量导致意外关机；
- **30°C~50°C 行完全一致**：锂电池在此温度范围内的电化学特性趋于稳定。

`Get_Battery_Capacity` 函数接受温度和电压两个输入，分别在行方向和列方向上进行就近匹配，定位到表格中最接近的单元格，返回对应的电量百分比。

**完整 SOC 估算流程**（由 `Task_Battery_Monitor` 编排）：

```
1. 调用 BSP_ADC_Process() → Ther_V（NTC电压）, Bat_V（电池电压）
2. 调用 Get_Temperature(Ther_V)  → 得到当前温度 current_temp
3. 调用 Get_Battery_Capacity(current_temp, Bat_V) → 得到 SOC 百分比
4. 根据 SOC 驱动 LED 指示灯：
   ├── SOC > 50%   → 亮绿灯、灭红灯（电量充足）
   ├── 20% ≤ SOC ≤ 50% → 全灭（省电模式）
   └── SOC < 20%   → 亮红灯、灭绿灯（低电量告警）
```

> **工程扩展性说明**：当前实现采用离散就近匹配，若后续需要更高精度，可升级为双线性插值（Bilinear Interpolation），在四个相邻节点间进行加权平均，算法复杂度仅增加常数项，适合在 Cortex-M 上实时运行。

#### 4.3.5 基于官方 Datasheet 的精准数据逆向建模

本系统中 NTC 温度查表数组与电量-温度二维矩阵的数值 **绝非经验估算或随意填写**，而是严格依据元器件官方数据手册（Datasheet）进行物理推导与工程建模后得到的精准标定值。此设计理念体现了从"能用"到"可信赖"的产品级工程素养。

##### （1）NTC 温度-电压 1D 数组的推导过程

系统选用 **航臣 10kΩ NTC 热敏电阻**，其官方规格书提供了完整的 **电阻-温度（R-T）对照表**，涵盖 -40°C 至 125°C 范围内每 5°C 步进的标称电阻值。

V2.0 版本的 NTC 检测电路采用 **上拉分压拓扑**：**1kΩ 固定电阻**从 VCC（3.3V）上拉，NTC 热敏电阻接地，ADC 采集中间节点电压。分压公式为：

$$
V_{\text{ADC}} = V_{\text{CC}} \times \frac{R_{\text{NTC}}(T)}{R_{\text{NTC}}(T) + R_{\text{fixed}}} = 3.3 \times \frac{R_{\text{NTC}}(T)}{R_{\text{NTC}}(T) + 1000} \quad (\text{V})
$$

将航臣官方 R-T 表中 -10°C 至 50°C 各温度节点的 NTC 电阻值代入上式，即可精确反推出对应的 ADC 分压基准值。以 20°C 为例：

$$
R_{\text{NTC}}(20°\text{C}) = 12.09\text{k}\Omega \quad \Rightarrow \quad V = 3.3 \times \frac{12090}{12090 + 1000} \approx 3.05\text{V} \approx 3.06\text{V}
$$

再如 -10°C 节点：

$$
R_{\text{NTC}}(-10°\text{C}) \approx 42\text{k}\Omega \quad \Rightarrow \quad V = 3.3 \times \frac{42000}{42000 + 1000} \approx 3.22\text{V} \approx 3.24\text{V}
$$

> 实际工程中，考虑到电阻精度公差（±1%）、PCB 走线阻抗及 ADC 基准电压偏移等因素，最终写入数组的数值在理论计算值基础上进行了小幅修正标定，确保实测温度与查表温度的偏差控制在 ±2°C 以内。

##### （2）V2.0 电路变更对测温精度的影响分析

V2.0 将 NTC 分压电路的固定电阻从旧版约 **12kΩ 降至 1kΩ**，这一硬件变更导致了电压查表数组的全面重建。两版电路的核心参数对比如下：

| 对比维度 | 旧版电路（V1.0） | 新版电路（V2.0） |
|:---|:---:|:---:|
| 固定电阻 R_fixed | ≈ 12kΩ | **1kΩ** |
| 电压范围（-10°C~50°C） | 0.60V ~ 2.80V | **2.58V ~ 3.24V** |
| 电压跨度 ΔV | ≈ 2.20V | **≈ 0.66V** |
| ADC 有效码数（12bit, ref=3.3V） | ≈ 2731 counts | **≈ 819 counts** |
| 每 °C 对应 ADC 码数 | ≈ 45.5 counts/°C | **≈ 13.7 counts/°C** |

**精度影响分析**：

- **电压分辨率下降**：新版电路在 60°C 温度跨度内仅产生 0.66V 的电压变化量，约为旧版的 30%。在相同 12bit ADC 下，温度分辨率从 ~45 counts/°C 降至 ~14 counts/°C，理论量化精度变差。

- **实际影响可控**：系统采用 **10°C 步进的就近匹配查表法**，相邻两个温度节点之间的 ADC 码数差约为 137 counts（13.7 counts/°C × 10°C），远大于 ADC 的 ±2 LSB 固有噪声。即使在最差情况下（ADC 随机抖动 ±5 counts），仍有超过 ±60 counts 的判定裕量，不会产生错误的温度跳变。

- **硬件简化收益**：1kΩ 电阻在 PCB 上更易获取标准值、体积更小、成本更低。同时，较小的固定电阻意味着分压网络在各温度下的电流变化幅度更小，对 NTC 的 **自热效应** 影响更为均匀，有利于减少非线性误差。

**建模意义**：相比于直接使用 NTC 的 Steinhart-Hart 方程进行实时浮点运算（涉及对数与倒数运算），预计算查表法将运行时开销降至纳秒级的数组索引操作，在保证精度的前提下极大节省了 CPU 时间片，这对于无 FPU 的 Cortex-M0+ 内核尤为重要。

##### （3）电量-温度 2D 矩阵的推导过程

系统配套的 **304060 型号聚合物锂离子电池**（标称容量 800mAh，标称电压 3.7V）的官方规格书提供了完整的多温度放电曲线族，包含不同温度条件下的 **容量衰减系数** 与 **截止电压特性**。关键物理参数摘录如下：

| 温度条件 | 放电容量保持率（相对 25°C） | 工程解读 |
|:---:|:---:|:---|
| **-20°C** | **≥ 60%** | 极寒环境下可用容量大幅衰减，满电 4.2V 实际仅可用约 480mAh |
| **0°C** | **≥ 80%** | 低温衰减显著，中低压段电压跌落加速 |
| **25°C（基准）** | **100%** | 标准测试条件，放电曲线最为平坦 |
| **60°C** | **≥ 95%** | 高温下活性物质反应充分，容量略有回升 |

二维矩阵的构建逻辑严格遵循上述衰减特性：

- **行维度（温度轴）映射**：将官方规格书的放电容量保持率折算为各温度行的电量天花板。例如，-10°C 行的 4.2V 列标定为 90%（而非 100%），即反映了该温度下满充电池的可用容量已衰减约 10%；

- **列维度（电压轴）映射**：提取官方放电曲线上的 **关键拐点电压**（4.2V 满电平台、4.0V 上平台尾端、3.8V 中段转折、3.6V 下平台起始、3.5V 急速跌落区、3.3V 截止保护电压），以非均匀间距设置列节点，在电压变化最剧烈的低压区（3.3V~3.6V）加密采样，提升低电量段的估算分辨率；

- **跨维度耦合**：低温行在中低电压列的电量值被大幅压低（如 -10°C / 4.0V 仅标定 50%，而 30°C / 4.0V 标定 90%），精确反映了 **低温下锂电池内阻急剧增大导致端电压"虚高"** 的核心物理现象——同样的端电压读数，在低温下对应的真实剩余容量远低于常温。

> **V2.0 标定修正**：相比旧版矩阵，V2.0 在 -10°C 和 0°C 两行的中低压段（3.6V~4.0V）进行了 **大幅度向下修正**（如 -10°C / 4.0V 从 70% 降至 50%，3.8V 从 40% 降至 15%）。这一修正基于最新批次电池在低温环境下的实测放电曲线——实测数据表明旧版标定对低温容量衰减过于乐观，导致冬季户外使用时电量指示偏高。V2.0 采取"宁低勿高"的保守策略，彻底消除低温下"电量从 40% 瞬间跳至关机"的用户痛点。

### 5.2 系统设计亮点

1. **算法鲁棒性设计**：心率检测采用"双极值提取 + 峰谷联合验证"架构，正峰法与负谷法互为冗余、交叉校验，任一路径受噪声干扰时另一路径自动兜底，显著提升了算法在不同佩戴条件和生理状态下的适用范围与工程可靠性。

2. **极致内存优化**：通过 Union 联合体将生命周期不重叠的采集/滤波缓冲区与寻峰缓冲区映射至同一物理内存，节省 2000 字节（占 8KB SRAM 的 25%），使复杂的 200 点批量信号处理算法得以在超低成本 MCU 上运行。

3. **分层校准抗噪机制**：心率通过静息校准锚点建立 → 偏差门控滤除跳变（ΔHR ≤ 10）→ 环形缓冲周期性刷新基准 → 滑动平均输出平滑的四步流水线实现精准去噪；血氧由于其比值计算本身稳定性较好，仅需较大窗口（=5）的滑动平均即可有效降噪。两条路径独立运行，消除了共用 index 推进逻辑可能导致的基准污染问题。

4. **工业级 BMS 思维**：电量估算未采用简单的电压线性映射，而是引入温度维度构建二维查表模型，V2.0 进一步基于实测数据修正了低温段标定值，体现了对锂电池电化学特性的深入理解，方案具备向量产产品迁移的工程基础。

5. **资源效率最大化**：全系统 Bare-Metal 架构，DMA 承担 ADC 采集和 UART 发送的数据搬运，CPU 专注于算法处理；定时器硬件重复计数器替代软件分频，所有设计选择均围绕"最小化 CPU 负载"展开。

### 5.3 后续优化方向

| 优化项 | 具体方案 |
|:---|:---|
| 峰谷检测精度提升 | 引入自适应阈值机制，根据近期 AC 幅值动态调整极值提取的候选筛选条件 |
| SpO2 标定精度 | 建立基于受试者样本的多段分段线性回归模型替代单一线性公式 |
| SOC 估算精度 | 升级为双线性插值；增加库仑计法（电流积分）作为辅助校准 |
| 低功耗优化 | 引入 STOP 模式休眠，仅由 RTC 唤醒执行采集任务 |
| 运动伪影抑制 | 增加加速度计（如 LIS3DH），通过自适应噪声对消算法分离运动分量 |
